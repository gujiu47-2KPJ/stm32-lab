# 开发日志 — FlashSystemDataProtect

> 项目基于 STM32F103C8T6 (Blue Pill)，实现断电保护按钮计数器。
> 开发环境：Keil MDK-ARM V5.32, STM32CubeMX 6.17.0

---

## 一、外设接线图（完整复现用）

### 硬件清单

| 器件 | 型号 | 数量 |
|------|------|------|
| 主控板 | STM32F103C8T6 (Blue Pill) | 1 |
| Flash 模块 | W25Q64 (SPI 接口) | 1 |
| OLED 屏幕 | SSD1306 0.96寸 (I2C 接口) | 1 |

## 二、最初版本的问题（项目最初代码的 Bug）

拿到这个项目的第一版代码时，里面有很多问题。下面按严重程度排列：

### 🔴 致命问题（导致程序跑不起来）

#### 1. OLED 显示行号/列号从 0 开始（实际需要从 1 开始）
- **文件**: `Core/Src/main.c` 中的 `OLED_Display_Status()` 函数
- **问题**: 调用了 `OLED_ShowString(0, 0, ...)`，但 `OLED_ShowChar()` 内部写的是 `uint8_t page_start = Line - 1`。当 Line=0 时，`0-1 = 255 (uint8_t溢出)`，导致数组越界写入，程序崩溃
- **现象**: OLED 闪一下就黑了，按 RESET 也没反应

#### 2. SPI 读 Flash ID 用了两次分开的发送/接收（丢掉第一个有效字节）
- **文件**: `Hardware/w25qxx.c` 中的 `W25QXX_ReadID()` 函数
- **问题**:
  ```c
  HAL_SPI_Transmit(... cmd, 4, ...);   // 发送4字节命令
  HAL_SPI_Receive(... data, 3, ...);   // 再接收3字节数据
  ```
  SPI 是全双工的——发送的同时也在接收。第一次 `Transmit` 发完 4 字节，W25Q64 已经把前几个 ID 字节输出到 MISO 上了，但这些数据被丢弃了。后面的 `Receive` 读取的是更后面的字节，全是 0x00。
- **现象**: 读到 Flash ID = 0x000000，与期望值 0xEF4017 不匹配，Flash 初始化失败

#### 3. 结构体有内存对齐填充（sizeof 比想象中大）
- **文件**: `Core/Src/main.c` 中的 `SystemData_TypeDef` 结构体
- **问题**:
  ```c
  typedef struct {
      uint8_t magic;         // 1字节
      uint32_t button_count; // 4字节 ← 需要4字节对齐
      uint8_t checksum;      // 1字节
  } SystemData_TypeDef;
  ```
  编译器为了让 `uint32_t` 对齐到 4 字节边界，在 `magic` 后面插了 3 个空白字节（padding）。实际占用 **9 字节**，不是以为的 6 字节。校验和计算时把 padding 里的随机值也算进去了，导致保存和读取算出来的校验和不一样，数据永远校验失败。
- **现象**: 断电后重新上电，计数归零，保存的数据读取不出来

### 🟡 中等问题

#### 4. 两个按键共用一个消抖计数器
- **文件**: `Core/Src/main.c` 中的 `Button_Process()` 函数
- **问题**: 两个按键共用一个 `debounce_counter`，如果一个键在抖动，另一个键的检测也会被干扰
- **后果**: 按键偶尔不灵敏

#### 5. 没有自动保存机制
- **文件**: `Core/Src/main.c` 中的按键处理
- **问题**: 原来是 PA0 作为"保存键"，要手动按 PA0 才存到 Flash；PA1 只是单纯计数不保存。用户想要的是按计数键就自动存
- **后果**: 断电前忘了按 PA0，数据就丢了

#### 6. OLED 刷新太慢，阻塞了按键检测
- **文件**: `Core/Src/main.c` 的 main 循环
- **问题**: 每轮循环都刷一次 OLED（通过 I2C 传 1024 字节数据到屏幕），这一下就要 200 多毫秒。加上 `HAL_Delay(50)`，一轮循环跑一次要 250ms 以上。按键消抖需要连续检测 4 次，你要按住按键超过 1 秒才能触发
- **后果**: 按键感觉非常不灵敏，像是没反应

#### 7. 重复的驱动文件
- **目录**: `FlashDriver/` 和 `Hardware/` 下有同名文件
- **问题**: 一个项目里有两套 W25QXX 驱动代码和两套 GPIO 初始化代码，里面实现细节不一样（有的加 `printf`，有的不加；有的加了唤醒延时，有的没加），维护起来很混乱

### 🟢 小毛病

#### 8. `W25QXX_PageWrite` 不是 static 函数
- 头文件里没声明它，但定义是全局可见的，容易和其他模块冲突

#### 9. 引用了不存在的头文件
- `FlashDriver/w25qxx.c` 里 `#include "usart.h"` 但项目里没有这个文件

#### 10. 文件末尾缺换行符
- 好几个源文件末尾没有空行，Keil 编译器会报警告

## 三、对最初版本的总评价

**一句话：文件结构混乱，核心功能有 Bug，没有数据保存逻辑。**

优点是基本思路是对的——知道要用 W25Q64 存数据、用 OLED 显示、用按键交互。真正的硬件接线也是对的。代码编译也能通过（0 Error），但烧录到板子上就跑不起来。

这很典型——对于刚学嵌入式的人来说，代码能编译通过不等于能跑。很多隐藏的问题要在硬件上实际测试才能发现。

## 四、第一次改了什么（让程序能跑起来）

### 改动 1：修复 OLED 行号从 0 变 1

**问题**: `OLED_ShowChar()` 内部用 `Line - 1` 索引数组，传 0 就变成 255，数组越界。

**解决**: 把原来的 `OLED_ShowString(0, 0, ...)` 改成 `OLED_ShowString(1, 1, ...)`。

OLED 屏幕内部是 8 个"页面"（page），每个字 8×16 像素占 2 个 page。所以有效行号是 1、3、5、7。

### 改动 2：修复 SPI 读 Flash ID

**问题**: `HAL_SPI_Transmit` + `HAL_SPI_Receive` 分开调用 = 丢掉有效数据。

**解决**: 改成用 `HAL_SPI_TransmitReceive` 一次性收发，然后取收到的第 2~4 字节作为 ID。

```c
// 原来（错误）
HAL_SPI_Transmit(&hspi1, cmd, 4, timeout);  // 4字节发完，收到的东西丢了
HAL_SPI_Receive(&hspi1, data, 3, timeout);  // 再读3个，读到的是无效的0

// 改成（正确）
HAL_SPI_TransmitReceive(&hspi1, cmd, rx, 4, timeout);
// rx[0] = 无效，rx[1] = 0xEF, rx[2] = 0x40, rx[3] = 0x17
id = (rx[1] << 16) | (rx[2] << 8) | rx[3];
```

### 改动 3：结构体加 __packed 去掉对齐填充

**问题**: 自然对齐导致 sizeof = 9，padding 字节的随机值破坏校验和。

**解决**: 在 `struct` 前面加 `__packed` 关键字，告诉编译器不要插空白字节，结构体紧凑到 6 字节。

### 改动 4：独立消抖 + 自动保存

**问题**: 两键共用一个消抖器 + 需要手按保存键。

**解决**: 
- 每个按键独立一个状态机（各自有自己的 `last_state` 和 `counter`）
- PA1 按下去后，计数 +1 并且**立刻自动保存到 Flash**
- PA0 改为"清零键"，按下去计数归零并保存

### 改动 5：用 dirty 标记减少 OLED 刷新次数

**问题**: 每轮循环刷一遍 OLED，一轮要 200+ 毫秒，按键检测被拖死。

**解决**: 加了一个 `display_dirty` 变量。只在按钮按下后才刷新 OLED，平时主循环只跑按键检测 + HAL_Delay(30)，30ms 一轮，按键响应快多了。

### 改动 6：清理冗余文件

删了 `FlashDriver/` 整个目录、`Hardware/gpio.c`、`Hardware/spi.c`、`Hardware/usart.c`。保留 `Hardware/w25qxx.c` 作为 Flash 驱动。

### 改动 7：修复 GPIO 配置（I2C 缺上拉 + SPI MISO 引脚模式）

在 `stm32f1xx_hal_msp.c` 中：
- I2C 的 PB6/PB7 补上 `GPIO_PULLUP`（否则 I2C 通信不稳定）
- SPI 的 PA6 (MISO) 从错误的 `AF_PP` 改回 `GPIO_MODE_INPUT`

## 五、用户提出需求后的迭代过程

### 第一版需求

用户说想要这样的功能：

> OLED 显示按钮按下次数，断电时自动保存到 Flash，重新上电后从上次的值继续累加。

我基于原来的代码进行了上述 7 个改动。这个阶段实现了：
- ✅ OLED 正常显示
- ✅ 按键能正常检测
- ✅ Flash 能正常读写
- ❌ **但断电后重新上电，计数全没了，显示为 0**

### 为什么丢了数据？

原因是**结构体对齐问题**（上面改动3）。`sizeof(SystemData_TypeDef)` 实际是 9 字节而不是 6 字节。保存的时候校验和算进去了 3 个随机字节，重新上电读回来校验和就对不上，系统以为 Flash 里没有有效数据，就从 0 开始了。

**解决**: 加 `__packed` 去掉对齐填充。

还有一个问题是首次上电时 Flash 是空的（全 0xFF），`Saved:` 行会显示 4,294,967,295 这个巨大的数字。在代码里加了个判断：如果数据无效，把 `saved_data.button_count` 手动清 0，避免显示垃圾值。

### 第二版需求

修复后用户测试通过。然后提出 UI 优化：

> 上电后 Saved 行显示上次断电时保存的值，按下按钮 Count 行累加但 Saved 行不变。只有下次断电再上电，Saved 才更新为最新的值。

也就是之前每次按按钮都会 `saved_data = current_data`，OLED 的 Saved 行跟着实时更新。用户不想要这个——Saved 行应该作为一个"参照线"不动，告诉你上次断电时记录是多少。等下次上电它才刷新。

**解决**: 删掉 `Save_Data_To_Flash()` 函数末尾的 `saved_data = current_data;` 这一句。

```
                      ┌───────────────────┐
                      │     上电启动       │
                      └───────┬───────────┘
                              │
                              ▼
              ┌─────────────────────────────┐
              │ 从 W25Q64 Flash 读取上次数据  │
              │ 校验 magic + 校验和           │
              └────────────┬────────────────┘
                           │
              ┌────────────┴────────────┐
              │ 校验通过?                │
              └────────────┬────────────┘
                   ┌──────┴──────┐
                   ▼              ▼
          ┌─────────────┐ ┌──────────────┐
          │ Count = 上次 │ │ Count = 0    │
          │ Saved = 上次 │ │ Saved = 0    │
          └──────┬──────┘ └──────┬───────┘
                 │               │
                 ▼               ▼
          ┌──────────────────────────────┐
          │      主循环 (while 1)         │
          │                               │
          │ 1. 检测 PA1 → Count+1 → 存到  │
          │    Flash（Saved 不变）         │
          │ 2. 检测 PA0 → Count清0 → 存到  │
          │    Flash（Saved 不变）         │
          │ 3. 只在 Count 变化时刷新 OLED   │
          │ 4. HAL_Delay(30)             │
          └──────────────────────────────┘
                     │
                     ▼
          ┌──────────────────────────────┐
          │ 断电 → Flash 保存最后的值     │
          │ 下次上电 → 回到流程开头       │
          └──────────────────────────────┘
```

```

     │    --Btn Counter--       │    ← 标题
```

| PA0  | 计数归零，自动保存到 Flash |

## 七、踩坑总结（给新手）

1. **OLED 驱动函数参数是 1-indexed 还是 0-indexed？** 看源码！`Line - 1` 和 `Column - 1` 就是 1-indexed
2. **SPI 是全双工的**，发送和接收同时发生。`HAL_SPI_Transmit` + `HAL_SPI_Receive` 分两步写，第一步收到的数据就丢了。要用 `HAL_SPI_TransmitReceive` 一次性收发
3. **结构体有内存对齐**，`sizeof()` 可能比你想象的大。Flash 存数据必须用 `__packed` 压缩
4. **按键消抖要独立**，两个按键不要共用一个计数器
5. **I2C OLED 刷新很慢**，每次传 1024 字节到屏幕要 200+ 毫秒。不要在每轮循环里都刷，用 dirty 标记只在数据变化时刷新
6. **代码编译通过 ≠ 能跑**，实际烧录到硬件上测试才能发现真正的问题
